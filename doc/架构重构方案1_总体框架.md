# Dong 架构重构（方案1）总体框架

> 最后更新：2026-02-04
> 
> 本文档是“方案1”的总体框架说明：明确 core/backends/app 的边界、注入点、关键契约与迁移方向。

---

## 0. 背景与目标

### 0.1 背景
Dong 是 GPU 加速 HTML/CSS 渲染引擎。当前代码历史包袱导致：
- core 层仍存在 SDL 依赖（include/类型/实现）
- backend 目录出现“公共模块被错误下沉”的情况（典型：GlyphAtlas）
- SDL 执行链存在双轨（`dong_sdl_execute.*` vs `sdl_gpu_driver*`），导致职责不清与后续优化困难

### 0.2 目标（方案1）
- **Core 完全去平台化**：core 不依赖 SDL（除 legacy 明确例外）
- **Backend 只承载平台绑定与 GPU 执行**：窗口/输入/设备/Shader/执行链集中在 backend
- **公共模块回归 core**：例如 `GlyphAtlas` 作为公共模块保留在 core，通过“最小 GPU 能力接口”隔离平台差异
- **能力注入而非下沉复制**：当公共模块需要 GPU 资源时，通过注入的能力（GPUTexture/GPUAtlas 等）工作，而不是把模块整体挪到某个 backend

### 0.3 非目标
- 不在本阶段追求一次性实现所有后端（Vulkan/D3D12 等），先以 SDL backend 做参照实现。
- 不在本阶段彻底移除 legacy API，但要把 legacy 与新架构隔离清晰。

## 1. 分层与职责

### 1.1 应用层（Applications）
- 负责加载 backend plugin（例如 SDL）
- 调用 `dong_engine_*` 或 `EngineView` 驱动 tick
- 处理窗口事件循环（通过 plugin poll_event 等）

### 1.2 Core（dong.dll）
- DOM/CSS/Layout/Script
- DisplayList 构建（`Painter`）
- GPU IR 生成（`GPUCompiler` 产出 `GPUCommandList`）
- 资源策略（例如 ResourceManager）
- 公共缓存/算法模块：
  - `GlyphAtlas`（MSDF 生成、cache/LRU、分页与装箱策略、metrics 等）

Core 的原则：
- **不包含 SDL 头文件**（除 legacy 明确例外）
- **不直接创建/操作 SDL 资源句柄**
- 通过“能力注入”访问 GPU/Atlas/FS/Logger/Decoder 等

### 1.3 Backend（dong_sdl_backend.dll 等）
- 窗口/输入/计时等平台绑定
- GPUDevice/Swapchain/Shader 管理
- GPUCommandList 的执行（pipeline、pass、提交、同步等）
- 可选的硬件/平台资源能力：视频解码、GPU 压缩转码等

Backend 的原则：
- 平台相关实现都在 backend
- 不复制公共模块（例如 GlyphAtlas 不应每 backend 各写一份）

## 2. 注入点与契约

### 2.1 Plugin API（应用层加载）
- `dong_plugin_api.h` 定义插件能力与 vtable（log/time/fs/window/input/renderer/video 等）
- 应用层加载 plugin，创建 vtable 与 user 指针，交给 core

### 2.2 Platform 单例（core 内依赖注入容器）
- `dong_platform.h` 提供 `dong_platform_set_* / get_*` 注入与访问
- 当前已存在：`DongGPUDriver`、`DongFileSystem`、`DongLogger`、`DongImageDecoder`、`DongSurfaceFactory`

方案1新增/强化的注入能力：
- **GPUTexture/GPUResource 能力**：用于 core 内管理普通纹理资源与后续 batch 优化
- **GPUAtlas 能力（AtlasFactory）**：core 通过工厂创建 `DongImageAtlas`，而不是直接调用某个 backend 的 create 函数

> 说明：`DongGPUDriver` 当前同时承担“资源能力 + 执行能力”。方案1建议在语义上区分，并在后续视情况拆分 ABI（不要求一次性完成）。

## 3. 关键公共模块：GlyphAtlas（回归 core）

### 3.1 为什么 GlyphAtlas 必须是公共模块
GlyphAtlas 包含：MSDF 生成、缓存策略、分页装箱、metrics 归一等，这些都与 SDL 无关。
若把它放入 `backends/sdl`，将导致：
- 每个 backend 都要复制并维护同一套逻辑
- 核心排版/渲染策略难以统一与回归测试

### 3.2 GlyphAtlas 的最小 GPU 依赖（通过能力注入）
GlyphAtlas 需要的 GPU 能力应尽量小：
- 创建/销毁 atlas 纹理页（2D RGBA8）
- 上传子矩形（支持 stride）
- 可选：返回 fence/ticket，用于异步上传后安全释放 staging

建议以 C 风格函数表注入（避免 C++ ABI 扩散）：
- `GlyphAtlasTextureOps`（概念名）

GlyphAtlas 不应依赖 `execute/prepare_resources` 等高层执行语义，避免在渲染执行阶段引入资源创建递归。

## 4. 图像 Atlas 与压缩（当前 CPU，未来 GPU TODO）

### 4.1 当前现状
- `DongImageAtlas`（C vtable）为平台无关接口
- SDL backend 提供实现 `dong_sdl_image_atlas_*`
- 当 atlas 格式为压缩（ASTC/BC）时，当前路径是：
  - RGBA8 → `DongImageDecoder.encode()`（CPU 生成压缩 blocks）→ `DongImageAtlas.upload()` 上传到压缩纹理

### 4.2 方案1的调整
- core 不直接依赖 SDL 的 atlas create：通过注入的 **AtlasFactory** 创建 `DongImageAtlas`
- `DongImageDecoder` 继续承担“解码/CPU 编码”能力（短期保留）

### 4.3 TODO：GPU 片元/Compute 压缩转码
目标：让压缩发生在 GPU 上，并直接写入目标压缩纹理/atlas，避免 CPU 生成 blocks 再上传。
建议把 GPU 压缩能力挂在“Atlas/Texture 上传路径”上：
- 例如新增 `upload_rgba8_transcode_to_compressed()`（可选 vtable）
- 或注入 `DongGPUCompressionOps`（由 backend 实现）

此 TODO 需要：
- shader/compute pipeline
- 压缩格式与 GPU 支持矩阵
- 同步与资源生命周期契约

## 5. 每帧数据流（渲染链）

1) DOM/CSS/Layout 更新
2) `Painter` 构建 DisplayList
3) `GPUCompiler` 生成 `GPUCommandList`
4) 资源准备阶段（建议固定在 execute 前）：
   - glyph 批量生成与上传
   - image 加载、可选压缩、上传到 atlas
   - video frame external image 更新
5) backend `execute(command_list)`：按命令流提交 GPU pass

核心约束：
- 资源创建/上传必须发生在“资源准备阶段”，避免在 execute 内部产生不可控递归与跨帧竞争。

## 6. 迁移策略（高层）

### 6.1 立即修正（阻断重复与双轨）
- GlyphAtlas 回归 core：删除 `backends/sdl/sdl_glyph_atlas.*`
- 退役 `backends/sdl/dong_sdl_execute.*`：SDL backend 的唯一执行链为 `sdl_gpu_driver*`

### 6.2 能力注入补齐
- 增加 AtlasFactory 注入
- 补齐 GPUTexture 上传 stride/ticket/fence 语义

### 6.3 验收标准
- core 无 SDL 头文件依赖（legacy 例外明确）
- backend 只承载平台绑定与执行链
- 多帧一致性与 baseline 对比回归通过（工具链见 `重构遗留.md`）
