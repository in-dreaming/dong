# Dong 引擎重构2：任务清单与验收标准

> 目的：把当前仓库的“重构缺口”拆成可交付的任务包（非 vibe），每项都有明确完成标准（可编译/可运行/可验证）。
> 
> 当前状态要点（基于代码 review）：
> - 已拆出三层：`dong`(core, SHARED, 不链接 SDL)、`dong_render`(STATIC, SDL 耦合渲染 + 旧 View)、`dong_appcore`(SHARED, SDL 窗口/事件/渲染封装，但内部仍用 `dong_view_*`)。
> - 新 `dong_engine_*` 已存在并能生成 `GPUCommandList`，但 demo/appcore 仍未切到该路径。
> - `backends/sdl` 已有 `dong_sdl_backend` 与 `Platform/GPUDriver/Surface` 注册，但 `execute()` 仍是 stub，未形成端到端渲染闭环。
> - `dong_platform.h` 里声明了 `DongFileSystem/DongLogger` 等，但缺少可用的公开接口定义与实现（对集成方不可用）。

---

## 0. 全局验收口径（适用于所有任务）

- **不允许“只改 demo 行数”当完成**：完成以“依赖关系与调用链”验收。
- **分层边界（必须满足）**：
  - `dong`（core）不直接依赖 SDL 头文件/符号、不编译 `src/render/sdl_render`、不包含 `src/platform/sdl3_window`、不包含 `src/input/input_adapter_sdl3`。
  - SDL 相关只允许出现在：`backends/sdl`、`plugins/sdl`、`appcore`、`examples`（示例层）。
- **最小可运行闭环（必须满足）**：至少一个 demo 走 `dong_engine_*` + `dong_sdl_backend`（或 plugin renderer）完整渲染出可见 UI。

---

## 1. 任务：明确公共 API 分层（头文件与 include 关系整改）

### 1.1 目标
把“core 对外 API”和“SDL 导向的 view API”彻底分开，避免调用方一包含 `dong.h` 就被迫接触 `dong_view.h` 的 SDL 语义。

### 1.2 具体事项
- 调整 `dong/include/dong.h` 的 include 策略：
  - 默认只暴露 `dong_engine_*`（以及 platform-agnostic 的基础类型）。
  - `dong_view.h` 不应默认被 `dong.h` include（改为显式 include 或单独聚合头）。
- 为 `dong_view.h` 明确定位：它属于“旧/SDL 耦合路径”还是“高级但仍平台无关”。若仍需要保留 SDL 版 view，则：
  - 要么将其移动/重命名为 `dong_view_sdl.h`（或放进 `dong_render` 的 include 安装目录），避免污染 core。

### 1.3 完成标准（验收）
- `examples/*` 中任何选择“新 API demo”的工程：
  - 只 include `dong.h` 就能工作，不需要被迫 include `dong_view.h`。
- `dong` 库的 public header 集合中，不出现 `SDL_*` 类型或 `#include <SDL...>`。

## 2. 任务：补齐 Platform 子系统的“正式化接口”（FS / Logger 等）

### 2.1 目标
`dong_platform.h` 宣称可注入的子系统，必须有可用的公开 C ABI 接口（struct/vtable + set/get + 默认行为）。

### 2.2 具体事项
- 定义并落地以下公共头文件（或整合进现有 include，但要求对外可用）：
  - `dong_file_system.h`：`DongFileSystem` 的 vtable（exists/read/read_partial 或 read_all）。
  - `dong_logger.h`：`DongLogger` 的 vtable（level + message + 可选 category）。
  - （可选）`dong_font_loader.h`：若短期不做，必须从 `Platform` 公开 API 中移除/延后，不允许“只有 forward declare”。
- 在 core 中把当前直接用 `std::ifstream` 等路径，迁移为通过 `DongFileSystem`（缺省实现可以是“只支持绝对路径 + stdio”，但必须定义清楚）。

### 2.3 完成标准（验收）
- `dong/include` 下所有在 `dong_platform.h` 暴露的类型都能在 include 层找到完整定义（无“只声明不定义”）。
- core 中加载资源（HTML 引用的 `img/src`、`script src`、CSS `url()` 等）可以通过注入的 FS 工作。
- 未注入 FS/Logger 时行为可预测：
  - FS：返回明确错误码或空内容；
  - Logger：默认为 no-op 或 stderr（需明确约定）。

## 3. 任务：建立 `dong_engine_*` 的 GPUDriver/Surface 渲染闭环

### 3.1 目标
让 `dong_engine_tick()` 产出的 `GPUCommandList` 真正通过 `DongGPUDriver->execute()` 跑起来（至少 SDL 后端），形成能渲染的闭环。

### 3.2 具体事项
- 明确 `GPUCommandList` 的“后端可执行格式”与版本管理：
  - 若后端需要 cast 到内部结构（当前 `dong_gpu_driver.h` 的注释允许），则必须提供：
    - 稳定的结构版本号；
    - 或者转换层（core 输出稳定 C 结构，后端执行）。
- 在 `backends/sdl/dong_sdl_platform.c` 中实现真正的 `execute()`：
  - 不能是 stub。
  - 可以复用现有 `src/render/sdl_render/*` 的执行逻辑，但需遵守分层：
    - SDL 执行代码应位于后端侧；
    - core 只产生命令，不包含 SDL。
- 明确资源生命周期：纹理/缓冲/管线谁创建、谁销毁、跨帧缓存策略、何时 invalidation。

### 3.3 完成标准（验收）
- 运行一个最小 demo（见任务 5）可以在窗口里看到 UI 渲染结果。
- `dong_sdl_backend` 的 `execute()` 不再是“永远 return 1”，而是：
  - 能处理至少：纹理创建/更新、绘制提交、present/离屏目标（按你当前命令集范围）。
- 多帧一致性工具链（`html_render_test --frames` / `run_multiframe_regress.py`）对至少一个用例输出一致（或差异可解释且有明确 issue）。

## 4. 任务：统一/收敛双套引擎逻辑（避免 EngineImpl 与 View 双维护）

### 4.1 目标
避免 `src/api_bindings.cpp` 的 `EngineImpl` 与 `src/core/view.cpp` 的 `View` 同时维护 DOM/Layout/Script/渲染逻辑，造成行为不一致。

### 4.2 具体事项
选择并执行一种收敛策略（必须明确选型，不能两头都保留）：
- **策略 A（推荐）**：以 `Engine` 为核心，`View` 变成 Engine 的一个薄包装（仅保留必要适配），最终 `dong_view_*` 只作为兼容层。
- **策略 B**：废弃 `EngineImpl`，让 `dong_engine_*` 内部复用 `View` 的能力（但要解决 View 的 SDL 依赖问题）。

### 4.3 完成标准（验收）
- DOM/Layout/Script 的主实现只有一套（或至少主路径只有一套）。
- 任何关键 bugfix 不需要在两个模块重复修改。

## 5. 任务：新增/改造一个“新 API 基准 demo”（必须走 `dong_engine_*`）

### 5.1 目标
提供一个 demo 作为后续所有改动的回归基准：它只使用 `dong_engine_*`（而不是 `dong_view_*`），并通过后端完成渲染。

### 5.2 具体事项
- 新增一个 example（建议：`examples/engine_minimal/` 或类似），或把现有 `examples/minimal/main.c` 改造为 engine 路线。
- demo 需要：
  - 创建窗口与 GPU 设备（SDL 层）；
  - 初始化 `dong_sdl_backend` 并注册到 `Platform`；
  - 创建 `dong_engine_t`，加载 HTML；
  - 每帧 `dong_engine_tick()` 并把输出呈现到 swapchain。

### 5.3 完成标准（验收）
- demo 源码中不出现 `dong_view_create/dong_view_render_to_gpu_texture/dong_view_render_offscreen`。
- demo 运行可见 UI。
- CMake/zig build 能一键构建运行（以你现有构建习惯为准：`zig build` 或 `zig build run-*`）。

---

## 6. 任务：AppCore 切换到新 API（或明确它是“旧链路封装”）

### 6.1 目标
AppCore 的定位必须与 `重构.md` 一致：要么成为“新 API 的高层封装”，要么明确是“旧 View 的封装并标注 legacy”。

### 6.2 具体事项
- 若 AppCore 要成为新 API：
  - `dong_app` 内部应创建/持有 `dong_engine_t`，而不是 `dong_view_t`。
  - 输入事件通过 `dong_engine_send_*` 路由。
  - 渲染通过 `dong_engine_get_command_list` + GPUDriver 执行，或通过后端统一 present。
- 若短期不切：
  - 必须改名/标注（例如 `dong_appcore_legacy`），并把新 AppCore 另起实现，避免“名义新、实际旧”。

### 6.3 完成标准（验收）
- `examples/interactive_demo_new.c`、`examples/minimal/main.c` 至少一个能完全不依赖 `dong_view_*`。
- `dong_app_get_renderer`、`dong_app_get_dong_engine` 等接口要么实现、要么从 public API 移除（不允许长期 `NULL // Not implemented yet`）。

---

## 7. 任务：清理/约束 legacy API 的边界（兼容层“可控存在”）

### 7.1 目标
保留 legacy 并非问题，但必须：
- 清晰边界（属于哪个库、link 哪个目标、给谁用）；
- 可编译开关能完整关闭；
- demo 不再默认走 legacy。

### 7.2 具体事项
- 为 legacy 提供统一开关（例如 `DONG_DISABLE_LEGACY_VIEW_API` 或现有宏的补全），并在 CI/本地脚本里至少跑一次“关闭 legacy”的构建。
- legacy 相关 header/符号尽量只由 `dong_render` 暴露/安装。

### 7.3 完成标准（验收）
- 关闭 legacy 后（只保留 engine API）仍可编译通过，并能运行至少一个 demo。
- `examples/` 默认构建不包含被注释的 legacy demos（目前 CMake 已注释，但要确保没有隐式依赖）。

---

## 8. 任务：渲染输出/Present 责任归属（避免双重 acquire/submit）

### 8.1 目标
当前存在“多个模块都可能 acquire swapchain / submit command buffer”的风险（例如 Scene3D 自己 submit，App 也可能 present）。需要明确：谁负责 present，其他人只能生成命令/子 pass。

### 8.2 具体事项
- 定义一个明确的帧边界 API（建议由后端或 AppCore 统一控制）：
  - begin_frame / execute / end_frame 的调用关系。
- 对 Scene3D/Overlay 等：只生成 draw 命令或使用统一的 renderer 接口，不直接在多个地方 acquire/sumbit。

### 8.3 完成标准（验收）
- 在 `3d_screens_simple` 路径中，不会出现“一个 frame acquire 两次/submit 两次”的结构性风险。
- 开启验证层/SDL debug 时不报 swapchain/acquire 相关错误（至少在 Windows 默认配置下）。

---

## 9. 任务：资源加载与路径规则统一（resource_root / 相对路径 / data 目录）

### 9.1 目标
把示例中的“各种候选路径探测”收敛到统一策略：资源根目录、相对路径解析、可执行目录等。

### 9.2 具体事项
- 规定：
  - `resource_root` 是一个目录；
  - HTML 中相对 URL 的解析规则；
  - `zig build` 下运行时 data 目录的定位方式。
- 将当前散落在示例/工具里的路径探测收敛为：
  - AppCore 统一负责（或
  - 提供一个 `examples/utils` 统一 helper）。

### 9.3 完成标准（验收）
- 任意 demo 在以下两种工作目录下都能找到 `examples/data`：
  - repo 根目录运行
  - `zig-out/bin` 运行
- `html_render_test`、`3d_screens_simple` 对资源路径的行为一致。

## 10. 任务：多帧一致性/闪烁排查的工程化接入（作为回归门槛）

### 10.1 目标
把你现有工具链变成“改渲染链路必须过的门槛”，避免后续偷懒导致回归。

### 10.2 具体事项
- 明确并固化两类回归：
  - **自一致性**：同一 HTML、同一参数，多帧输出一致（`run_multiframe_regress.py`）。
  - **基线对比**：与浏览器 baseline 的 diff 在可接受范围（`run_baseline_compare.py`）。
- 给出最小用例集（例如 3~5 个 HTML case），每次关键改动必须跑过。

### 10.3 完成标准（验收）
- 新 API 渲染闭环完成后，上述脚本至少在 1 个 case 上可跑通并产出结果文件（OK 或明确标注 nondet）。

---

## 11. 任务：构建目标与依赖关系清理（CMake/Zig 的“真分层”）

### 11.1 目标
确保每个 target 的 link 依赖符合分层，不出现 core 被动链接 SDL 的风险。

### 11.2 具体事项
- `dong`：不得链接 SDL3/SDL_shadercross/dxcompiler。
- `dong_sdl_backend`：只链接 SDL3 + `dong`（允许）。
- `dong_appcore`：若切新 API，依赖应从 `dong_render` 迁移到 `dong` + `dong_sdl_backend`。
- examples：按 demo 路线选择 link 到正确的库。

### 11.3 完成标准（验收）
- `CMakeLists.txt` 中每个 target 的依赖关系与注释一致。
- 可以单独构建 `dong` 而不需要 SDL 相关 third-party 可用（除非你明确要求所有依赖一体化）。
